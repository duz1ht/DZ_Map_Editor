Reading these files suggests a split between logical definitions near the top of the file and concrete placed items later in the file. Events then reference those logical numbers through compact trigger/action rows.

1. Waypoints

What the files show clearly

  • Every map contains logical waypoint definitions with begin waypoint N.
  • Placed entities can carry waypoint_id and wpdistance.
  • Waypoint marker items also exist as regular items named "Waypoint" on layer 8 in many maps.
  • Some waypoint definitions are clearly special markers, such as Goto SSN, Goto Group and Goto Player in ADK_02a.mis.
Safe conclusion: waypoints are not just a visual editor helper. They are a numbered logical system that items and events can reference.

Observed mental model

Logical waypoint definition
        ↓
Placed marker or placed AI entity references waypoint_id
        ↓
wpdistance defines reach radius / tolerance
        ↓
Mission logic and AI can react to that numbered waypoint

How to read it

begin waypoint 125 defines a logical slot. The descriptive name shows that not every waypoint is a simple patrol point.

The placed item named "Waypoint" gives that waypoint a concrete world position through position X Y Z. In this export style, positions are stored as integers that correspond to the engine fixed-point world coordinates.

When an AI unit or another mission system references a waypoint_id, it is almost certainly referring to one of these numbered logical nodes.

Important caution: do not confuse logical waypoint definitions with every item that happens to carry a waypoint_id. The files show both the abstract definition layer and the placed object layer.

2. Area Triggers

What the files show clearly

  • Every logical area is declared by begin areatrig N.
  • Many maps contain placed items named "Area Trigger" on layer 8.
  • Those placed items carry a wpnumber that matches the logical area number.
  • Across several maps, the number of placed "Area Trigger" items is exactly 2x the number of logical areas.
Strong observed pattern: one logical area is usually represented by two placed Area Trigger markers sharing the same number. That strongly suggests opposite corners or paired bounds used to build a spatial region.

How to read it

The logical declaration begin areatrig 5 defines the existence of area 5. The placed marker item later in the file places one corner or bound marker for that area through position and wpnumber 1, wpnumber 2, and so on.

Because the files repeatedly show two Area Trigger items for each logical area, the safest working model for a compatible editor is this:

Logical area N
   = two placed Area Trigger markers
   = same wpnumber N
   = world positions define the region

That model fits the file evidence very well, even if the runtime later stores or converts the region in a different internal form.

3. Layers

What the files show clearly

  • Maps declare layers explicitly with blocks such as begin layer 1, begin layer 8, and so on.
  • Placed items carry a numeric layer field.
  • The top comments also summarize per-layer counts, for example Total Layer 8 : 748 (Markers) in SPBHD_16.mis.
  • Common names include Foliage, Decorations, Buildings, Vehicles, Objects, Power-Ups and Markers.

Best interpretation

Layers look like an editor organization system. They are extremely useful in the file, but nothing in the event blocks suggests that layer numbers are themselves mission triggers.

For building a compatible editor, treat layers as:

visibility bucket
selection bucket
editing bucket
category label

  Color "18"
end layer

4. Groups

What the files show clearly

  • Maps declare groups with begin group N.
  • Placed items carry group_id.
  • Group 1 and Group 2 are often Blue Players and Red Players.
  • Events very often reference numbers that match group-oriented mission behavior.
Safe conclusion: groups are the main logical bucket for controlling several entities together.

Useful editor model

Group definition
      ↓
Many entities share the same group_id
      ↓
Events operate on the group number
      ↓
Whole squads / waves / sets can be enabled,
disabled, moved, checked or affected together

How to read it

The item example shows a real world object carrying both group_id 19 and waypoint_id 19. This is a good example of how these systems overlap instead of existing in isolation.

A helicopter, AI unit or mission object can belong to a group and also be tied to waypoint logic. That is exactly the kind of cross-reference a mission editor must preserve.

Best practice for your editor: store group membership on the entity itself, not only in a separate group table. That matches how the .mis data is actually written.

  A mission event can target the group
  The entity can still navigate or progress via waypoint data

5. Events

What the files show clearly

  • Maps define events with begin event N.
  • Events may include delay_value.
  • Events may include one or more rows inside begin triggers.
  • Events may include one or more rows inside begin actions.
  • The raw trigger/action rows are compact numeric tables. They clearly reference other numbered mission systems.

Best interpretation

Event
 ├ id
 ├ optional delay
 ├ trigger rows
 └ action rows

Trigger rows decide when the event becomes true.
Action rows define what the map changes when it fires.

Even without fully decoding every opcode, the structure is unambiguous.

Compact event example

begin event 3
  name "Event 3"
  begin triggers 1
    0 1 7 3 3 10 -1 0
  end triggers
  begin actions 1
    0 1 3 4 -1 -1 0 0
  end actions
end event

How to read it

Event 3 in ADK_02a.mis shows the smallest useful pattern: one trigger row and one action row.

Event 4 in SPBHD_16.mis shows a richer pattern: one trigger, an explicit delay, and many action rows affecting several numbered targets.

That repeated file structure strongly supports this editor design:

Event object
  id
  name
  delay
  triggers[]
  actions[]

Even if you do not decode every numeric field immediately, preserving row order and values is crucial for a faithful editor.

  IF trigger row condition becomes true

Typical mission chain inferred from the analyzed maps

1. Player crosses a numbered Area Trigger
2. An event trigger row for that area becomes true
3. The event actions activate or affect a Group
4. Entities in that Group move, react or become relevant
5. Waypoint-linked progression continues the mission
6. Another event may fire later and continue the sequence

What is safe to implement in another editor

Waypoints

  • Store logical waypoint definitions by number.
  • Store placed waypoint marker items with world position.
  • Preserve waypoint_id and wpdistance.

Area Triggers

  • Store logical areas by number.
  • Store two placed markers per area when present.
  • Preserve wpnumber and marker positions exactly.

Layers

  • Store layer definitions and names.
  • Store item layer membership.
  • Treat layer as editor organization, not mission logic.

Groups

  • Store logical group definitions by number.
  • Store group_id on each entity.
  • Allow events to target groups without flattening them away.

Events

  • Preserve event id, name and delay.
  • Preserve every trigger row exactly.
  • Preserve every action row exactly.

Position

  • Preserve raw integer position X Y Z values.
  • Handle them as engine-compatible world coordinates.
  • Avoid converting away precision on save.

Final takeaway

The analyzed .mis files support a consistent and useful mental model:

Items are the placed things in the world.
Layers organize those items in the editor.
Groups let multiple items behave as one logical set.
Waypoints provide numbered navigation / progression references.
Area Triggers provide numbered spatial regions.
Events hold the trigger rows and action rows that make the mission move.

The strongest file-backed insight is that these systems are connected by numeric references. A faithful editor should therefore preserve not just the visible objects, but also the numbered logical tables and the links between them.
