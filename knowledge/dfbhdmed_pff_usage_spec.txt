PFF ARCHIVE USAGE AS OBSERVED IN DFBHDMED.EXE

Scope and evidence constraints
This document describes only behaviors that are directly supported by artifacts observable inside the executable. Statements are anchored to concrete string evidence extracted from the binary. Any behavior that cannot be directly tied to such evidence is marked UNKNOWN.

PFF concept presence and dedicated subsystem
The executable contains explicit internal identifiers and log labels that treat PFF as a first class container format. The string 'pffmgr' appears as an internal component identifier, and there are dedicated log labels 'PFF ENTRIES', 'PFF STRUCT', and 'PFF LOADED FILE'. These strings indicate a subsystem responsible for parsing a PFF container, exposing a list of entries, and reporting loaded resources as originating from a PFF archive. Evidence: strings at offsets 0x000B9D6C='PFF LOADED FILE: %s', 0x000B9DF0='PFF ENTRIES', 0x000B9DFC='PFF STRUCT', 0x000B9E08='PFF LOADED FILE', 0x000B9E20='Error reading %s in PFF file %s.' and 0x000B9E18='pffmgr'.

Unified file access layer and PFF integration
Near the PFF log labels there is a compact cluster of file abstraction names that appear to wrap OS file operations: 'file_Create', 'file_OpenEx', 'file_FileSizeEx', 'file_LoadFileEx', and 'file_RawLoaderEx'. In the same cluster the executable also contains the generic log label 'LOADED FILE: %s' and the PFF specific log label 'PFF LOADED FILE: %s', plus the error message 'Cannot open file %s' and the PFF specific error message 'Error reading %s in PFF file %s.'. This co-location supports the conclusion that the editor uses a shared file loading pathway that can resolve resources either as regular files or as entries inside a PFF archive, while preserving the origin in logging. The exact resolution order between filesystem and PFF is UNKNOWN. Evidence offsets: 0x000B9D48='_filelog.txt', 0x000B9D58='LOADED FILE: %s', 0x000B9D6C='PFF LOADED FILE: %s', 0x000B9D84='file_Create', 0x000B9D90='LOADED FILE', 0x000B9D9C='file_OpenEx', 0x000B9DA8='file_FileSizeEx', 0x000B9DB8='file_LoadFileEx', 0x000B9DC8='file_RawLoaderEx', 0x000B9DDC='Cannot open file %s', 0x000B9DF0='PFF ENTRIES', 0x000B9DFC='PFF STRUCT', 0x000B9E08='PFF LOADED FILE', 0x000B9E18='pffmgr', 0x000B9E20='Error reading %s in PFF file %s.'

Known PFF archives referenced by name
The executable embeds a fixed set of PFF archive file names. These are 'exp4.pff', 'exp3.pff', 'exp2.pff', 'exp1.pff', 'updated.pff', 'localres.pff', 'resource.pff', and 'med.pff'. The names are stored contiguously and appear inside the same data region as terrain tiler and tile manager strings, which strongly indicates they are used as a search set for resource resolution in map editing features (at minimum the tiler). Whether all of these archives are mandatory, optional, or used only in specific editor modes is UNKNOWN. Evidence offsets: 0x000AAA08='exp4.pff', 0x000AAA18='exp3.pff', 0x000AAA28='exp2.pff', 0x000AAA38='exp1.pff', 0x000AAA48='updated.pff', 0x000AAA58='localres.pff', 0x000AAA68='resource.pff', 0x000AAA78='med.pff'

Tile manager and tiler functionality linked to PFF archives
The strings 'tile manager', 'tile image', and 'tile image bits' appear in the same compact region as the embedded PFF archive names. The same region also includes 'CANTFINDFILE', the source reference-like string 'maptile.c ln:289', and a user-facing warning stating that the maximum number of tiles has been reached and no more tiles may be placed. Additional tiler UI/control identifiers appear nearby, including 'Tiler', 'TOOLS_TILER', 'TILE_SHOW', 'CLEAR_THIS_TILESET', 'Tiler Warning', 'CLEAR_THIS_MISSION', and 'FILE_CLEAR'. The proximity of these strings supports that the tiler is the feature that most directly depends on resources loaded through PFF archives, and that it uses PFF resolution for tile images or tile related data. The specific entry names requested from PFF archives by the tiler are only partially evidenced. Tile image naming evidence relevant to PFF lookups
The executable contains the wildcard-like tile image pattern string 'trntile*.tga'. This is direct evidence that tile images may be referenced by a naming convention beginning with 'trntile' and stored as TGA images. It is UNKNOWN whether the tiler enumerates these images from a PFF directory listing or derives individual names from map metadata. Evidence offsets: 0x000A7954='trntile*.tga'

MIS field names that plausibly drive tile resource resolution
The executable contains the MIS key name 'terrain_tile_tga' both as a bare key token and as a formatted line pattern '  terrain_tile_tga %s'. This is evidence that the map file can specify a tile texture or tile sheet name via a field that the editor reads and writes. Whether the value of this field is resolved as a loose file, as a PFF entry, or through a combined mechanism is UNKNOWN, but the presence of the PFF archive set in the tiler region supports that PFF-based resolution is at least one supported mechanism. Evidence offsets: 0x000ADBE8='terrain_tile_tga', 0x000AE644='  terrain_tile_tga %s'

Terrain map resource identifiers
The executable contains internal identifiers that name three distinct terrain related maps: 'polytrn_colormap', 'polytrn_depthmap', and 'polytrn_detailmap'. These strings are evidence that the editor expects to load or process at least these three map layers for the terrain system. The executable does not, in the extracted string corpus, embed a direct file extension for these map identifiers nor does it embed the term 'heightmap'. Therefore the mapping between 'depthmap' and any height representation is UNKNOWN. It is also UNKNOWN whether these maps are stored inside PFF archives or loaded as standalone files, because no direct string evidence links these identifiers to a PFF path or extension. Evidence offsets: 0x000AB3C0='polytrn_tilesinfo', 0x000AB3F4='polytrn_tilestrip', 0x000B43C4='polytrn_origin', 0x000B43D4='polytrn_sectors', 0x000B43E4='polytrn_sectorcount', 0x000B43F8='polytrn_scale', 0x000B4408='polytrn_wrapy', 0x000B4418='polytrn_wrapx', 0x000B4428='polytrn_tileinfo', 0x000B443C='polytrn_polydata', 0x000B4450='polytrn_depthmap', 0x000B4464='polytrn_detailmap', 0x000B4478='polytrn_colormap'

Observed error and validation behaviors specifically related to PFF
There is a dedicated PFF read error message 'Error reading %s in PFF file %s.'. This indicates that the editor attempts to read a named entry from a named PFF container and can surface a read failure. The executable also contains the generic file open failure message 'Cannot open file %s'. It is UNKNOWN whether 'Cannot open file %s' is used for both loose files and PFF containers, or only for loose file operations. Evidence offsets: 0x000B9DDC='Cannot open file %s', 0x000B9E20='Error reading %s in PFF file %s.'

Logging and trace artifacts
The file name '_filelog.txt' appears adjacent to 'LOADED FILE: %s' and the PFF specific 'PFF LOADED FILE: %s'. This is evidence that the editor can log file load operations and distinguishes between standard loads and PFF-origin loads. The runtime conditions under which this logging is enabled, the exact formatting of a log record, and whether the log includes resolved paths are UNKNOWN. Evidence offsets: 0x000B9D48='_filelog.txt', 0x000B9D58='LOADED FILE: %s', 0x000B9D6C='PFF LOADED FILE: %s', 0x000B9D90='LOADED FILE', 0x000B9E08='PFF LOADED FILE'

Per-archive usage details that are UNKNOWN with current evidence
The executable embeds multiple archive names, but the string evidence does not include explicit per-archive role annotations such as 'terrain in exp?.pff' or 'ui in med.pff'. Therefore the following are UNKNOWN and cannot be proven from the current extracted evidence alone: which specific resources are pulled from each archive; the priority order among exp1 through exp4, updated, localres, resource, and med; whether the archives are opened at startup or on demand; whether missing archives are tolerated; and whether the editor ever writes or modifies PFF archives. The only proven behaviors are that a PFF subsystem exists, that it reports entries and structure, that it logs PFF-origin loads, and that the tiler/tile manager region embeds the archive name set alongside tile related strings.

Artifacts that would be required to elevate UNKNOWN items to proven behavior
To produce a per-archive, per-resource, and per-operation description of what the editor does with PFF files, additional derived artifacts from the same executable are required. The minimal set is: disassembly excerpts or decompiled pseudocode for the routines that open PFF containers and resolve a requested resource name; cross references from each embedded archive name string to its usage site; cross references from 'trntile*.tga', 'terrain_tile_tga', and 'polytrn_*' identifiers to code that forms a load request; and, if available, a runtime log capture that includes '_filelog.txt' output showing which entries are loaded from which PFF file when opening or editing a map. Without such artifacts, any attempt to attribute resources to specific PFF archives would be speculation and must remain UNKNOWN.
