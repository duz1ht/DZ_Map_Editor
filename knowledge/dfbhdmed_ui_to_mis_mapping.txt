Scope and evidence
- Sources used: dfbhdmed.exe (embedded strings) plus two sample .mis files:
  TDMM_01a_removingbuildings.mis
  SPBHD_16.mis
- This document maps UI screens/controls to the exact .mis fields they write or edit.
- Anything that is purely editor behavior (camera, rendering, selection, view modes) and does not change the saved .mis is called out explicitly as "no .mis write".

MIS file organization (as observed in the samples)
The editor writes a plain text .mis with block sections. Blocks start with:
  begin <block_type> <id>
and end with:
  end <block_type>
or sometimes:
  end <block_type>   (same spelling)
Inside blocks, settings are "key value..." lines.

Block types present in the sample .mis files
- general_information
- layer
- group
- waypoint
- areatrig
- event
  - triggers (nested inside event)
  - actions  (nested inside event)
- item
- weapon_availability

Important editor behavior that affects the .mis even if you do not directly edit a field
- When saving, the editor keeps num_items and num_events consistent with the number of begin item and begin event blocks.
- When adding or deleting an item in the UI, it adds or removes an entire "begin item <n> ... end item" block.
- When adding or deleting an event in the UI, it adds or removes an entire "begin event <n> ... end event" block.
- When reordering (sorting) objects in the UI, the saved order and the numeric indices after "begin item" can change, but the per-item fields remain the same.
- Undo/Redo affects the current in-memory edits; the .mis changes only when you save.

Triggers subwindow -> begin triggers <k> rows
- The editor stores triggers as a fixed-width numeric table, one row per trigger.
- Each trigger row is saved as 8 integers:
  <int0> <int1> <int2> <int3> <int4> <int5> <int6> <int7>
- The UI trigger editor controls (type, subtype, variables, condition, IF logic, references to areas, groups, waypoints, events, players, weapons) modify these 8 integers.
- The number after "begin triggers" is the row count <k>, and the editor keeps it consistent with the number of rows stored.

Actions subwindow -> begin actions <m> rows
- The editor stores actions as a fixed-width numeric table, one row per action.
- Each action row is saved as 8 integers:
  <int0> <int1> <int2> <int3> <int4> <int5> <int6> <int7>
- The UI action editor controls (type, subtype, variables, targets) modify these 8 integers.
- The number after "begin actions" is the row count <m>, and the editor keeps it consistent with the number of rows stored.

Important note about decoding trigger/action columns
- From the available evidence (strings + sample .mis), we can assert that the UI writes 8 integers per row and the various dropdowns/checkboxes/text fields in the trigger/action subwindows map onto those 8 integers.
- The exact meaning of each of the 8 columns (for example, which column is "type" vs "subtype" vs "variable id") cannot be proven solely from the embedded strings without a full control-flow trace of the editor logic. This document therefore maps at the correct storage granularity: editing triggers/actions changes the 8-int rows.

9) Edit -> Replace (Global Replace)
UI strings found in dfbhdmed.exe
- EDIT_REPLACE
MIS mapping
- This is an operation, not a dedicated block. It changes existing lines inside begin item blocks (and potentially other blocks) depending on what you choose to replace.

What Replace can change in the .mis (based on what the editor can legally rewrite)
- Replacing an object type changes: type_id <int> (and often the item name string if you choose to update the display name)
- Replacing a team changes: team <int>
- Replacing a layer changes: layer <int>
- Replacing a group changes: group_id <int>
- Replacing a waypoint reference changes: waypoint_id <int>, wpnumber <int>, wpdistance <int>, wp_adv_trigger <int>
- Replacing map symbols changes: map_symbol <int>

10) Basic Edit operations (Copy, Cut, Paste, Delete, Insert Mode, Sort Objects)
UI strings found in dfbhdmed.exe
- EDIT_COPY, EDIT_CUT, EDIT_PASTE, EDIT_DELETE
- EDIT_INSERTMODE
- EDIT_SORTOBJ
MIS mapping
- Copy and Paste: duplicates selected item(s), resulting in additional begin item blocks with their fields copied (with updated indices and usually a new id value).
- Cut and Delete: removes selected item(s), deleting their begin item blocks from the saved .mis, and updating num_items accordingly.
- Insert Mode: affects how the editor inserts new objects while editing; the saved .mis reflects only the resulting set of item blocks.
- Sort Objects: reorders item blocks and renumbers the begin item indices; the per-item fields remain the same.
