DFBHD Mission Editor UI Effects on .MIS (What the UI Actually Writes)
Generated: 2026-02-23 (America/Sao_Paulo)

Sources
- dfbhdmed_ui_to_mis_mapping.txt (UI strings and observed .mis writes)
- BMS_plaintext_spec.txt (plain-text rules and bitmask/enums derived from bms.h + bms.cpp)

Goal
Describe, with no omissions, which dfbhdmed.exe UI controls change the saved .mis file, and exactly what they change.

General rule: when a UI action is marked as "no .mis write" it only affects editor state (camera, view filters, selection, rendering). The .mis changes only after Save, and only via the fields described below.

--------------------------------------------------------------------
A) .MIS structure written by the editor
--------------------------------------------------------------------
The editor saves a plain-text .mis composed of blocks:
  begin <block_type> [index_or_id]
    key value...
  end <block_type>

Block types observed (from dfbhdmed_ui_to_mis_mapping.txt samples):
- general_information
- layer
- group
- waypoint
- areatrig
- event (with nested triggers and actions tables)
- item
- weapon_availability

Editor-maintained counts (written into general_information):
- num_items is updated to match the number of begin item blocks on save.
- num_events is updated to match the number of begin event blocks on save.

6) Edit -> Events

Menu path: Edit -> Events
Blocks: begin event <n> ... end event, with nested tables

Event header lines (mission dependent):
- name "<string>"
- reset_value <int> (optional)
- delay_value <int> (optional)
- attrib <int> (optional bitmask; event_flags in BMS spec, see Appendix 3)

Triggers table
- Stored as: begin triggers <k> ... end triggers
- Each trigger row is exactly 8 integers:
  condition_flags main_type sub_type param1 param2 param3 param4 unknown7
  This corresponds 1:1 to the Trigger struct in BMS_plaintext_spec.txt.
- UI edits in the Triggers subwindow update one or more of these 8 integers per row.

Actions table
- Stored as: begin actions <m> ... end actions
- Each action row is exactly 8 integers:
  reserved0 action_type action_sub_type param1 param2 param3 param4 reserved1
  This corresponds 1:1 to the Action struct in BMS_plaintext_spec.txt.
- UI edits in the Actions subwindow update one or more of these 8 integers per row.

Type/subtype enums (what the dropdowns mean in storage terms)
- Trigger main types and trigger subtypes are stored as integers (see Appendix 4).
- Action types and their subtypes are stored as integers (see Appendix 5).

Counts
- The <k> after begin triggers is kept consistent with the number of trigger rows.
- The <m> after begin actions is kept consistent with the number of action rows.

9) Edit -> Global Replace

Menu path: Edit -> Global Replace
This is an operation, not its own block. It rewrites existing lines inside blocks.

Observed/allowed replacements and the .mis effect:
- Object type -> type_id <int> (and optionally name "<string>" if you also update display name)
- Team -> team <int>
- Layer -> layer <int>
- Group -> group_id <int>
- Waypoint reference -> waypoint_id <int>, wpnumber <int>, wpdistance <int>, wp_adv_trigger <int>
- Map symbol -> map_symbol <int>

10) Basic Edit operations

Menu path: Edit -> Copy / Cut / Paste / Delete / Insert Mode / Sort Objects

Copy/Paste
- Duplicates selected item(s), producing new begin item blocks with copied fields.
- The begin item indices may change, and new id values may be assigned depending on editor behavior.

Cut/Delete
- Removes selected item(s), deleting their begin item blocks, and updating num_items on save.

Insert Mode
- Affects insertion workflow; the .mis only reflects the final set of item blocks after save.

Sort Objects
- Reorders begin item blocks and may renumber the begin item indices; per-item fields are unchanged.
