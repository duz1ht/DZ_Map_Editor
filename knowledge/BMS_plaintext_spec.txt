BMS parser logic rewritten as plain-text oriented rules
Source of truth: bms.h + bms.cpp provided by user.

1) High level file layout
The binary file is parsed and written in the exact order below (see parse() and write() in bms.cpp):
  A) Header (fixed size 616 bytes)
  B) Weapon loadout chunk (byte blob, length = header.weapon_loadout_chunk_len)
  C) Entities, in four consecutive arrays:
     1. items: header.num_items entries
     2. buildings: header.num_buildings entries
     3. markers: header.num_markers entries
     4. organics: header.num_people entries
  D) Waypoint records: exactly 128 records (fixed size 136 bytes each)
  E) Group records: exactly 64 records (fixed size 32 bytes each, treated as raw bytes)
  F) Layer records: exactly 32 records (fixed size 20 bytes each, treated as raw bytes)
  G) Area triggers: header.area_trigger_count records (fixed size 32 bytes each)
  H) Counts: events_count (i32), trigger_count (i32), action_count (i32)
  I) Events: events_count records (fixed size 24 bytes each)
  J) Triggers: trigger_count records (fixed size 32 bytes each)
  K) Actions: action_count records (fixed size 32 bytes each)
  L) Bounding boxes: bounding_box_count (i32) then that many records (fixed size 36 bytes each)

If parsing finishes and Reader.position() != file size, the parser fails with 'Did not consume all data'.

2) Primitive encoding rules that matter for plain text
2.1 Fixed strings
  read_fixed_string(dst, N) reads exactly N bytes, copies up to the first 0 byte into dst, and ensures dst is null-terminated.
  write_fixed_string(src, N) writes N bytes: src bytes until 0 byte or length limit, then pads with 0.
  Plain text implication: these become normal quoted strings, but are length-capped to N-1 visible bytes when round-tripping through BMS.

2.2 Fixed-point 16.16
  read_fixed16() reads i32 v and returns v / 65536.0f.
  write_fixed16(f) writes i32(round(f * 65536)).
  Plain text implication: any field documented as fixed 16.16 can be represented as float in text, but BMS stores i32 fixed-point.

2.3 Integers and floats
  u8/i8/u16/i16/u32/i32 are read and written little-endian.
  f32 uses IEEE-754 float (Reader.read_f32 / Writer.write_f32).

7.2 WaypointRecord marker list and padding rules
  marker_count is validated: if marker_count > 32, parsing fails.
  waypoint_numbers is a list of marker_count u32 entries.
  After reading the list, the record always consumes 128 bytes reserved for numbers, so it reads remaining padding bytes: (128 - marker_count*4).
  Plain text implication: represent waypoint_numbers as a variable-length list, but preserve marker_count and do not exceed 32.

8) Group records (fixed 64) and Layer records (fixed 32)
The parser treats GroupRecord and LayerRecord as raw byte blocks; it does not interpret their internal fields.
GroupRecord struct fields (in serialized order):

LayerRecord struct fields (in serialized order):
  raw_data[kLayerRecordSize]: uint8_t

Plain text implication: if you want a lossless round-trip using only this code as source of truth, you must store these records as opaque byte arrays in text (for example hex) or keep them untouched from the original BMS.

9) Area triggers
AreaTrigger struct fields (in serialized order):
  wp_number: int32_t
  min_z: int32_t min_x, min_y,
  max_z: int32_t max_x, max_y,
  reserved: int32_t

9.1 Coordinate swap rule
When parsing AreaTrigger, the file stores Y and Z swapped for both min and max corners.
  File order: min_x, file_min_y, file_min_z then max_x, file_max_y, file_max_z.
  In memory: min_x, min_y = file_min_z, min_z = file_min_y; same for max.
When writing AreaTrigger, the swap is reversed to match the file layout.

Plain text implication: store area trigger boxes in normal X,Y,Z order; when converting to BMS apply the Y/Z swap exactly as above.

10.2 Event reset_after and delay packing
The file stores reset_after and delay as i32 values where the upper 10 bits contain the value and the lower 22 bits are zero.
  On parse: reset_after = (raw_i32 >> 22), delay = (raw_i32 >> 22).
  On write: raw_i32 = (reset_after << 22), raw_i32 = (delay << 22).
Plain text implication: represent reset_after and delay as normal integers (0..1023 if you want to be faithful to 10 bits).

14) How to compute and edit flags in plain text (generic logic for all bitmasks in this code)
  To add a flag: value = value | FLAG_VALUE
  To remove a flag: value = value & (~FLAG_VALUE)
  To test a flag: (value & FLAG_VALUE) != 0
  When the value is 0, no flags are enabled.

15) Validation and failure conditions present in code
  - is_bms() requires magic 'BMS' in the first 3 bytes of header.magic, otherwise parse fails.
  - parse_header() requires the header to be exactly 616 bytes consumed, otherwise parse fails.
  - parse_entity() requires each entity record to be exactly 0xAC bytes consumed, otherwise parse fails.
  - parse_waypoint_record() fails if marker_count > 32, and also checks record size 136 bytes.
  - At the end of parse(), the reader position must equal file size, otherwise parse fails.
  - write() requires waypoint_records.size() == 128, group_records.size() == 64, layer_records.size() == 32, otherwise write fails.

16) Notes for converting BMS to a plain text format
If your goal is a text format that can round-trip through this exact code without losing information, these parts must be represented explicitly:
  - Header fields including unknown* bytes and values
  - GroupRecord raw_data and LayerRecord raw_data (opaque bytes)
  - WeaponLoadout raw_data (opaque bytes)
  - BoundingBox.unknown_data (opaque bytes)
  - Any Entity unk* fields (still stored in the Entity struct)
If you do not need lossless round-trip, you can omit unknown fields, but then writing back through this code will not reproduce the original binary exactly.
